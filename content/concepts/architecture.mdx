## End-to-end flow

### Session creation (one-time)

1. **Browser (SDK)** — `createSession({ cap, expirationSeconds, ... })` is called.
2. **`GET /supported`** — SDK reads the facilitator pubkey (`spender`) and asset metadata.
3. **Simulate `approve_checked`** — SDK builds a bare `IncoToken::approve_checked(encryptedCap)` instruction and simulates it to extract the `delegated_amount` handle from the user's IncoAccount, then derives the allowance PDA = `[handle_le16, facilitator_pubkey]`.
4. **Sign message** — User signs the canonical session JSON `{user, spender, asset, recipient, cap, expirationUnix, network, nonce}` with `signMessage`.
5. **Sign transaction** — User signs the final `approve_checked` tx (with allowance PDA in `remaining_accounts`) with `signTransaction`. The facilitator is the `feePayer` — user pays no SOL.
6. **`POST /sessions`** — SDK posts `{ approveTxBase64, authMessage, authSignature, ... }`. Facilitator verifies the Ed25519 signature, cross-checks every field, submits the approve tx (acting as fee payer), and inserts a sqlite row.

### Per-call settlement

7. **`session.fetch("/api/whatever")`** — wraps `fetch`; first call gets `402` with `PAYMENT-REQUIRED`.
8. **Retry with `PAYMENT-SIGNATURE`** — SDK base64-encodes `{ sessionId, amount, accepted, resource, x402Version }` and re-sends the request.
9. **Resource server forwards** the header to **`POST /settle`**.
10. **Facilitator** — debits sqlite atomically (`spent + amount ≤ cap`, not expired), builds an `IncoToken::transfer(amount_ciphertext)` tx (creating the recipient ATA if missing), simulates to get fresh source + dest amount handles, derives both decrypt-allowance PDAs, then signs as fee payer and submits.
11. **On-chain** — IncoToken CPIs into Inco Lightning; the Covalidator (Inco's TEE) performs `recipient.amount += amount` and `user.amount -= amount` over encrypted `Euint128` handles, signs the result, and writes new handles back.
12. **Response** — `{ success: true, transaction: <signature>, extensions.session: { spent, remaining } }`. Resource server returns the real handler response with a `PAYMENT-RESPONSE` header.

## Components

| Layer | Role |
|-------|------|
| **Client SDK** (`solana-x402-sessions`) | `createSession`, `wrapFetch`, resource-server scheme plugin |
| **Facilitator (Express)** | `/sessions`, `/verify`, `/settle`, `/supported`, `/pay/*`. Session state in sqlite. Fee payer for every submitted tx. |
| **IncoToken (Anchor)** | `approve_checked`, `transfer_with_authorization`, ATA management, Ed25519-bound auth path |
| **Inco Lightning + Covalidator** | FHE ciphertext arithmetic in a TEE; produces signed updates to encrypted balance handles |

## Trust model recap

- **On-chain enforced** by IncoToken `approve` allowance PDA: total cap and expiry. `transfer_with_authorization` checks `e_ge(allowance, amount)` via Inco Lightning before debiting.
- **Off-chain enforced** by the facilitator: per-call amount, recipient binding, session bookkeeping. The atomic sqlite debit + refund-on-failure keeps the in-flight cap consistent across crashes.
- **Privacy**: amounts and balances are `Euint128` ciphertext handles on chain — only the holder of the relevant allowance PDA can ask the Covalidator to decrypt.
